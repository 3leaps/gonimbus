# Gonimbus v0.2.1 Release Notes

**Release Date**: 2026-05-22
**Status**: Reflow Destination Metadata — Explicit Control, Explicit Disclosure Discipline

## Overview

Gonimbus v0.2.1 is a focused release for operators who use `transfer reflow` to land data into destination buckets or local filesystems and need the destination objects to carry durable metadata.

The release has four chapters:

1. **Destination metadata controls** — explicit user metadata, source metadata preserve/merge policies, content-type preservation, storage-class assignment, and metadata-aware provider PUT paths.
2. **Per-object metadata derivation** — allow-listed source metadata copy and source-derived expressions evaluated per object, with a small scalar-only v1 expression language.
3. **Security and disclosure discipline** — destination metadata disclosure warnings, allow-list posture, destination-system-metadata non-goal, and redacted derivation failures.
4. **Release-lane hardening** — exact Go toolchain pinning, SBOM/vulnerability artifacts, and YAML formatter pins.

For the comprehensive list of changes, see [CHANGELOG.md](../../CHANGELOG.md#021---2026-05-22).

## Chapter 1: Destination metadata controls

### The shift

Before v0.2.1, `transfer reflow` copied object bytes and rewrote destination keys, but destination metadata was mostly left to provider defaults. That was safe and simple, but it forced operators who needed durable destination attributes to build side pipelines or post-processing steps.

v0.2.1 keeps the old default and adds explicit controls. If you do not pass metadata flags, reflow still writes destination objects without caller-controlled user metadata, without source content-type preservation, and without explicit storage-class assignment.

### Explicit destination user metadata

```bash
gonimbus transfer reflow --stdin \
  --dest 's3://dest/landing/' \
  --rewrite-from '{key}' \
  --rewrite-to '{business_date}/{key}' \
  --metadata-policy clear \
  --metadata-set dataset=transactions \
  --metadata-set owner=data-platform
```

`--metadata-set key=value` is repeatable. Keys are normalized to lower case. If a key is repeated, the last value wins.

### Source metadata policies

| Policy     | Behavior                                                                                         |
| ---------- | ------------------------------------------------------------------------------------------------ |
| `clear`    | Do not copy source user metadata; only explicit `--metadata-set` and derivation outputs are used |
| `preserve` | Copy source user metadata; fail the object if source keys collide after lower-case normalization |
| `merge`    | Copy source user metadata, then apply `--metadata-set` overrides                                 |

Preserve and merge use the same canonicalization guard as per-object derivation. If a source metadata map contains keys such as `Foo` and `foo`, gonimbus does not guess which value should win.

### Content-Type and storage class

```bash
gonimbus transfer reflow --stdin \
  --dest 's3://dest/landing/' \
  --rewrite-from '{key}' \
  --rewrite-to '{key}' \
  --preserve-content-type \
  --destination-storage-class STANDARD_IA
```

`--preserve-content-type` copies the source object's content type. `--destination-storage-class <class>` writes an explicit destination storage class. `--destination-storage-class propagate` copies the source storage class, treating an empty source class as `STANDARD`.

Gonimbus accepts PUT-target classes such as `STANDARD`, `INTELLIGENT_TIERING`, `STANDARD_IA`, `ONEZONE_IA`, `GLACIER_IR`, and `REDUCED_REDUNDANCY`. Archive classes that are not valid PUT targets are rejected.

### Provider behavior

S3 and file providers now expose metadata-aware PUT paths. These compose with the default conditional reflow path, so `--on-collision skip-if-duplicate` still uses `PutObjectConditionalWithOptions` where the destination supports it.

Local file destinations cannot store object metadata natively, so gonimbus writes a cleartext JSON metadata sidecar next to the file. The default suffix is `.gnb-meta.json` and can be changed with `--metadata-sidecar-suffix`.

## Chapter 2: Per-object metadata derivation

### The shift

Many reflow jobs need each destination object to carry metadata derived from the corresponding source object: source lineage, source system fields, or a named subfield inside an upstream metadata blob. v0.2.1 adds that without widening provider interfaces and without projecting whole source metadata maps by default.

### Source-key projection

```bash
--metadata-set-from-source-key source-md5=md5
```

This copies one source user-metadata key (`md5`) into one destination user-metadata key (`source-md5`) for each object.

### Expression-derived metadata

```bash
--metadata-set-from-source-derived source-etag='system.etag'
--metadata-set-from-source-derived source-modified='system.last_modified'
--metadata-set-from-source-derived broker-device='urldecode(meta.payload).device'
--metadata-set-from-source-derived source-tag='system.etag + "-src"'
```

Supported v1 expressions:

| Expression family                | Example                          | Output contract                                             |
| -------------------------------- | -------------------------------- | ----------------------------------------------------------- |
| JSON subfield                    | `meta.payload.device`            | Named scalar subfield from source user metadata             |
| URL-decoded JSON subfield        | `urldecode(meta.payload).device` | URL-decode first, then read a named scalar subfield         |
| Source ETag                      | `system.etag`                    | Gonimbus `ObjectMeta.ETag`; S3 is cleaned/unquoted          |
| Source last modified             | `system.last_modified`           | UTC RFC3339Nano                                             |
| Source content length            | `system.content_length`          | Base-10 byte count                                          |
| Source content type              | `system.content_type`            | Source content type string                                  |
| Source storage class             | `system.storage_class`           | Source storage class; empty source normalizes to `STANDARD` |
| One string-literal concatenation | `system.etag + "-src"`           | Left expression plus right string literal                   |

JSON strings render verbatim. JSON numbers render as base-10 strings while preserving source precision; v0.2.1 uses `json.Decoder.UseNumber`-style handling rather than float64 round-tripping. JSON booleans render as `true` or `false`.

### Missing-source policy

```bash
--metadata-on-missing-source skip  # default
--metadata-on-missing-source empty
--metadata-on-missing-source fail
```

This policy covers absent source keys, missing JSON subfields, invalid JSON, invalid URL escapes, decoded-invalid JSON, null values, and unsupported non-scalar JSON results.

- `skip` omits only the affected destination metadata key.
- `empty` writes an empty string for the affected destination key.
- `fail` emits a redacted per-object `gonimbus.error.v1`.

When `fail` is paired with `--on-collision quarantine --collision-quarantine-prefix <prefix>`, gonimbus still emits the redacted error record and then files the object under the quarantine prefix with `status=quarantined reason=metadata.derivation.quarantined`. The quarantine copy uses empty metadata options so a failed or partial derivation does not get written onto the quarantine object.

## Chapter 3: Security and disclosure discipline

### Destination metadata is durable

Destination metadata is not a log message and is not a private transient value. It becomes part of the destination object's observable state.

The following are visible to destination readers with HEAD/GET or filesystem access and are not redacted at destination:

- Values supplied with `--metadata-set`
- Values copied by `--metadata-policy preserve|merge`
- Values copied by `--metadata-set-from-source-key`
- Values derived by `--metadata-set-from-source-derived`
- Content-Type preserved with `--preserve-content-type`
- StorageClass written or propagated with `--destination-storage-class`
- File-provider metadata sidecars (`.gnb-meta.json` by default)

If source metadata might contain credential URIs, tokens, account identifiers, or other sensitive values, use `--metadata-policy clear` and declare exactly the destination keys you intend to publish.

### Per-object derivation is an allow-list

Gonimbus rejects wildcard destination keys and wildcard source subfield projection. There is no "copy all subfields" mode in v0.2.1.

This matters because source-side metadata blobs often contain more fields than destination consumers need. Operators should inventory the source metadata shape and declare one destination key per intended output.

### Scalar-only output protects the boundary

`--metadata-set-from-source-derived` writes only scalar JSON subfield values:

- string
- number
- boolean

JSON `null` is treated as missing. Arrays and objects are unsupported non-scalar results and route through `--metadata-on-missing-source`. This prevents an operator from accidentally serializing an entire nested JSON object that contains fields they did not name in the allow-list.

### Destination system metadata remains store-owned

The new per-object derivation flags read source-side `system.<field>` values as inputs, but they write only destination user metadata.

For S3-compatible stores, a rule such as:

```bash
--metadata-set-from-source-derived source-etag='system.etag'
```

creates a destination user-metadata key such as `x-amz-meta-source-etag`. It does not override the destination object's actual ETag. Gonimbus does not set destination ETag, destination LastModified, destination ContentLength, or other object-store-generated fields.

Content-Type and StorageClass are separate because object stores expose them as writable PUT attributes. Gonimbus writes them only through `--preserve-content-type` and `--destination-storage-class`.

### Failure paths redact source values

Derivation-evaluation failures name the destination key, expression, and reason kind, but do not echo raw source metadata values in error JSONL, checkpoint rows, or default stderr. Covered failure classes include invalid JSON, invalid URL escapes, decoded-invalid JSON, non-scalar fail mode, and source canonical-collision failures.

## Chapter 4: Release lane hardening

### Exact Go patch pin

CI and release workflows now pin Go `1.25.10`. Vulnerability scanners attribute standard-library findings to the Go toolchain used to build or scan the project, so release evidence needs an exact patched toolchain rather than a floating `1.25.x` selector.

### Stable dependency evidence

`make dependencies` now emits stable SBOM and vulnerability-report artifacts under `sbom/`:

- `sbom/gonimbus.cdx.json`
- `sbom/gonimbus-vuln.json`
- `sbom/gonimbus-dependencies.json`
- `sbom/go-version.txt`

`sbom/` remains ignored in normal working trees; release jobs collect the artifacts explicitly.

### Format and database flakes

`.yamlfmt` and `.yamllint` pin YAML formatting behavior so local `make fmt` and CI agree. Index-store setup now applies SQLite `busy_timeout` before WAL mode to reduce intermittent root DB open lock failures.

## Upgrade notes

### Operator action

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.2.1
```

Existing reflow commands keep their old behavior unless you opt into the new metadata flags. Before adopting the new flags in a pipeline, review:

1. Which destination principals can read object metadata.
2. Which source metadata fields may contain sensitive values.
3. Whether `--metadata-policy clear` plus explicit allow-list rules is the right posture.
4. Whether missing or malformed source metadata should `skip`, write `empty`, `fail`, or `fail` into a quarantine route.

### Compatibility

No JSONL schema breaking change is introduced in v0.2.1. The new metadata options are opt-in. `--on-collision log` remains deprecated from v0.2.0; use `--on-collision skip-if-duplicate`.
