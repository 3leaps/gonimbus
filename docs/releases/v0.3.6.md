# Gonimbus v0.3.6 Release Notes

**Release Date**: 2026-07-04
**Status**: Incremental Index Top-Ups and Query-Time Deltas

Gonimbus v0.3.6 focuses on recurring index operations. Operators with stable
index manifests can now stop paying full enumeration cost on every top-up when
their layout supports date-partition scope narrowing, and downstream consumers
can ask the current index for objects first seen or changed after a completed
run.

The release has two user-facing parts:

- `index build --since <timestamp>|auto` narrows what Gonimbus lists when the
  manifest has a date-partitioned scope, then reports whether that reduction
  actually applied.
- `index query --since-run <run_id>` emits a forward delta over current active
  rows, so consumers can process only objects added or meaningfully changed
  after a known successful run.

Point-in-time historical views are intentionally not part of this release. The
current index stores latest object state, not snapshots for every prior run, so
`--at-run` remains deferred to a future index-substrate release.

## Highlights

- **Incremental listing with `--since`** - recurring builds can use an explicit
  timestamp or `--since auto` to derive a safe lower bound from the latest
  successful run in the same IndexSet.
- **Reduction signal, not silent optimism** - since builds report whether
  enumeration reduction was applied, partially applied, or unavailable, and
  emit per-prefix `added`, `changed`, and `unchanged` counts.
- **Query-time forward deltas** - `index query --since-run <run_id>` returns
  current active rows first seen or changed after a successful run in the same
  IndexSet.
- **Schema-v8 delta metadata** - object rows now persist first-seen and
  last-changed run IDs so the query delta is derived from stored index facts.
- **Fail-closed boundary checks** - unknown, failed, cross-IndexSet, or
  pre-migration boundary runs are rejected instead of producing an ambiguous
  delta.

## Incremental builds with `--since`

`index build --since <timestamp>` is for recurring top-ups that should ingest
objects modified at or after a known lower bound:

```bash
gonimbus index build --job index-manifest.yaml --since 2026-07-02T00:00:00Z
```

`--since auto` reads the latest successful run for the same IndexSet and uses
that run's start time as the lower bound:

```bash
gonimbus index build --job index-manifest.yaml --since auto
```

Bare `--since` is shorthand for `--since auto`.

When the manifest uses `build.scope.type: date_partitions`, Gonimbus can narrow
the date range before provider LIST calls. That is the cost-reduction path: the
provider receives fewer list prefixes before object metadata is filtered and
stored. For mixed `union` scopes, the run can report partial reduction when
only some children can be narrowed. For layouts that cannot map the watermark
to a narrower prefix plan, Gonimbus falls back to full enumeration plus a
last-modified ingest filter and reports that reduction was not applied.

That signal is part of the operator contract. Do not assume a since run was
cheaper because it succeeded; check the since-plan output and the per-prefix
delta report.

## Current-state deltas with `--since-run`

`index query --since-run <run_id>` reads the current index state and emits rows
first seen or meaningfully changed after the boundary run:

```bash
gonimbus index query \
  --index-set idx_da038d8171b4a9ba \
  --since-run run_1783087200000000000
```

The output remains `gonimbus.index.object.v1`. Delta queries add optional
fields such as `change_kind`, `first_seen_run_id`, and `last_changed_run_id`.
`change_kind` is `added` when the object first appeared after the boundary and
`changed` when a previously known or reappeared object changed after the
boundary.

This is a forward delta over latest state, not history. The current index does
not retain object snapshots for older runs, so it cannot answer "what did the
index look like at run X?" and does not ship `--at-run`.

Deletion history is also outside this query surface. `--include-deleted
--since-run` is rejected because the index cannot infer deletion deltas from
latest-state rows.

## Steady-state cadence

A typical recurring pattern is:

1. Keep one stable manifest per operational shard.
2. Run frequent `index build --since auto` top-ups for active shards.
3. Confirm the run reported enumeration reduction where expected.
4. Use `index query --since-run <run_id>` to feed downstream consumers with
   current objects that were added or changed after a known boundary.
5. Schedule periodic full-coverage audit builds when deletion detection
   matters.

Since builds are not full-coverage audits. They do not prove that objects
outside the narrowed listing plan still exist, so they skip soft-delete and can
leave deleted objects visible until a full audit build runs.

## Compatibility and internal storage

Index database timestamp storage is now normalized to a fixed-width UTC text
form that is preserved by both the default SQLite driver and the optional
`gonimbus_libsql` build. CLI and JSON output timestamps remain unchanged.

Users who query `index.db` directly may see internal timestamp fields stored
with a `+0000 UTC` suffix instead of `Z`. Existing RFC3339/RFC3339Nano stored
values remain readable and are migrated automatically.

Schema v8 adds `first_seen_run_id` and `last_changed_run_id` to object rows.
Older indexes migrate forward automatically. For delta correctness, precise
added/changed classification starts at the migration baseline; `--since-run`
rejects older boundary runs instead of fabricating a confident result.

## Upgrade notes

- Existing full index builds and normal `index query` filters remain
  compatible with v0.3.5.
- Add `--since auto` only after confirming the manifest identity is stable for
  the shard you intend to top up.
- Treat since-plan output as required evidence that enumeration reduction
  applied. If reduction is unavailable, the run can still succeed, but it paid
  the full listing cost.
- Keep periodic full-coverage audit builds when deletion detection matters.
- Use `--since-run` for "process new or changed current objects" flows, not for
  point-in-time reconstruction.

- A bounded field shakedown validates that `--since` reduction is observable on
