# Gonimbus v0.3.7 Release Notes

**Release Date**: 2026-07-05
**Status**: Operational Data Root Overrides

Gonimbus v0.3.7 is a focused release for operators who need predictable
placement of local operational state. Index databases, index-build job records,
and operation checkpoints now resolve through one app-wide data root, so a
deployment can move local state to durable storage without changing each command
surface independently.

The preferred one-process override is:

```bash
GONIMBUS_DATA_DIR=/mnt/gonimbus-data gonimbus index build --job index.yaml
```

For persistent configuration, set:

```yaml
data_root: /mnt/gonimbus-data
```

`GONIMBUS_DATA_ROOT` and `data_dir` remain accepted as compatibility aliases.

## Highlights

- **One operational data root** - indexes, index-build job metadata, server job
  defaults, and operation checkpoints now share the same resolver.
- **Environment and config overrides** - `GONIMBUS_DATA_DIR` wins for a single
  process, followed by `GONIMBUS_DATA_ROOT`, `data_root`, `data_dir`, and then
  the platform app-data default.
- **Doctor visibility** - `gonimbus doctor` reports the resolved data root, the
  source that selected it, and whether Gonimbus can write or create it.
- **Repository-local guardrail** - resolved data roots inside a git working tree
  are rejected before state is created, regardless of source, including
  symlink-resolved paths.
- **Owner-only state permissions** - app-data directories and runtime state files
  are created with owner-only permissions where Gonimbus controls them.

## Data root precedence

Gonimbus resolves the operational data root in this order:

1. `GONIMBUS_DATA_DIR`
2. `GONIMBUS_DATA_ROOT`
3. `data_root` in config
4. `data_dir` in config
5. `XDG_DATA_HOME` through the platform config layer
6. Platform app-data default

The resolved root is normalized and symlink-resolved before policy checks run.
That means an apparently external path that points back into a repository is
still rejected.

## No automatic migration

Changing the data root changes where future operational state is read and
written. Gonimbus does not copy existing indexes, job records, or operation
checkpoints between roots automatically.

Move existing state deliberately during maintenance windows, then run
`gonimbus doctor` and a small read/query check before resuming production jobs.

## Storage guidance

Use storage that behaves like a local filesystem for the operational data root.
Indexes and checkpoints are active runtime state; placing them on high-latency
or weakly consistent network mounts can make long-running jobs slower or less
predictable.

Do not place the data root inside a repository checkout. That keeps generated
operational state out of the public source tree and prevents accidental
inclusion in commits, archives, or review diffs.

## Compatibility

- Existing command flags and manifest formats are unchanged.
- Existing default behavior remains automatic when no data-root override is set.
- `GONIMBUS_DATA_ROOT` and `data_dir` remain supported aliases, but new
  automation should prefer `GONIMBUS_DATA_DIR` and `data_root`.
- No Stable library API changes are included in this release.

## Upgrade notes

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.3.7
```

Operators who previously relied on command-specific local defaults should set
`GONIMBUS_DATA_DIR` or `data_root` before starting new long-running jobs, then
verify the selected root with `gonimbus doctor`.
