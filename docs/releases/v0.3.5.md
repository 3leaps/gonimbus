# Gonimbus v0.3.5 Release Notes

**Release Date**: 2026-07-01
**Status**: Multipart Export/Reflow and Migrated Reflow Engine

Gonimbus v0.3.5 focuses on the paths that matter once object storage work gets
large: exporting an index hub, reflowing large objects, and keeping long-running
reorganization runs moving through transient provider pressure. `index export`
and `transfer reflow` now share a multipart upload primitive, so large writes can
cross the >5 GiB single-PUT boundary. The release also completes the v0.3.4
`pkg/reflow` migration for the stdin reflow subset, making that CLI path run
through the Experimental embeddable engine instead of a CLI-only implementation.

## Highlights

- **Multipart upload for large writes** - `index export` and `transfer reflow`
  use a shared multipart primitive for S3-compatible destinations when writes
  cross the configured threshold.
- **Large index hub exports** - a large `index.db` can be exported to an
  S3-compatible hub instead of failing at the provider's single-PUT limit.
- **Large-object reflow** - `transfer reflow` can copy large objects through
  multipart upload while preserving collision policy, metadata, storage class,
  provenance, and adaptive concurrency behavior.
- **Migrated reflow engine (Experimental)** - stdin reflow records for the
  migrated subset now route through `pkg/reflow`.
- **Error-path hardening** - content-probe terminal errors preserve their code
  and sanitized message, and throttled reflow probes retry before falling back to
  per-object failure handling.

## Multipart writes for export and reflow

The main operator headline is that large writes now have a common multipart path.
`index export` and `transfer reflow` both use the shared `pkg/transfer` upload
primitive for S3-compatible destinations. When a write crosses the multipart
threshold, Gonimbus spools bounded parts to local disk, uploads them as multipart
parts, completes the upload at the end, and aborts the upload on failure paths it
controls.

For `index export`, this means a large hub artifact such as `index.db` can be
published to a remote hub. For `transfer reflow`, it means a large source object
can land at the destination through the same safety posture used for smaller
objects: collision policy still applies, metadata and storage-class settings are
still honored, and provenance/summary records still describe what happened.

Multipart ETags remain provider-specific. Gonimbus treats multipart-form ETags
as non-comparable for byte-equality proof unless another safe comparison path is
available, so the new upload path does not turn an ETag shape into blind
overwrite permission.

## The reflow engine migration continues

v0.3.4 introduced `pkg/reflow` as an Experimental surface for the reflow engine's
records, adaptive-concurrency substrate, and redaction helpers. v0.3.5 moves the
migrated stdin reflow subset onto that engine for metadata planning, dry-run
planning, record-stream copy execution, collision decisions, adaptive
concurrency, and typed run / summary records.

The CLI remains the primary supported interface. Existing stdin reflow workflows
should behave as before, while unsupported forms continue to use the legacy path
until later migration work. For Go embedders, the important status is unchanged:
`pkg/reflow` is still **Experimental** under
[`docs/api-stability.md`](../api-stability.md). There are no Stable library API
breaks in this release.

## GCS-shaped reflow capability coverage

The release adds GCS-shaped capability fixtures for the reflow engine. This does
not change the provider matrix from v0.3.4; it verifies that the provider-neutral
gating in the migrated engine handles GCS destination semantics such as
conditional IfAbsent and unavailable ETag `If-Match` writes.

## Error-path hardening

Two long-run error paths were tightened:

- `content probe` now recognizes `gonimbus.error.v1` records as already-terminal
  input. It re-emits them without creating a new `INTERNAL` wrapper and
  preserves retryable `TRANSIENT` codes.
- Content-probe error output now uses the same provider-error sanitizer as
  reflow, and parser diagnostics no longer carry raw input JSONL lines forward.
- Reflow probe operations now retry throttled HEAD/body-compare attempts through
  the adaptive limiter. If throttling never clears, the existing per-object
  failure behavior remains; the checkpoint/resume classifier boundary is
  unchanged.

## Upgrade notes

- Existing CLI workflows remain compatible with v0.3.4.
- Large multipart writes need local scratch space for part spooling. Size that
  space against concurrent writes and configured part size.
- Configure the destination provider's lifecycle cleanup for incomplete
  multipart uploads, such as S3 `AbortIncompleteMultipartUpload` lifecycle
  rules. Gonimbus aborts uploads on the failure paths it controls, but lifecycle
  cleanup is the operator backstop for interrupted processes or provider-side
  partial state.
- `pkg/reflow` remains Experimental. Consumers should pin versions and expect
  the surface to evolve until it is promoted after downstream embedder use.

## Verification

- Unit, focused reflow, cloud-integration, and release quality gates are run
  before the release tag.
- A bounded real-cloud, bring-your-own-bucket pre-tag validation checks a >5 GiB
  multipart `index export` and high-parallel reflow path. The validation uses
  environment-supplied bucket parameters, skips when unset, and cleans up created
  objects on exit.
