# Gonimbus v0.3.0 Release Notes

**Release Date**: 2026-06-12
**Status**: Pure-Go Index Defaults, Resumable Operations, Provider-Dispatched Reflow, and Release-Gate Hardening

Gonimbus v0.3.0 is a minor release for operators and Go embedders running
large object-store workflows. It removes the default CGO dependency from local
index stores, adds resumable failure handling for long-running index and reflow
operations, makes local-tree-to-object-store reflow a first-class path, and
adds clearer failure classes for retry orchestration.

## Pure-Go Index Store by Default

Default builds now use the pure-Go `modernc.org/sqlite` driver for local index
stores, which simplifies static, cross-compiled, and container builds (no C
toolchain required). Local SQLite-backed index workflows continue to work
without CGO:

```bash
gonimbus index init
gonimbus index build --job crawl-manifest.yaml
gonimbus index query 's3://bucket/path/**/*.xml'
```

The libsql/Turso driver remains available for operators who need remote libsql
databases:

```bash
go build -tags gonimbus_libsql ./cmd/gonimbus
```

The release also refreshes the SQLite substrate dependencies and keeps a
compile-graph fixture in CI so storage-free library packages do not accidentally
pull in index-store dependencies.

## Resumable Long-Running Operations

Long-running `index build`, `index enrich-with-head`, and `transfer reflow`
runs now have a failed-resumable path for runtime interruptions. When a
resumable runtime failure occurs, gonimbus writes a sensitive-local operation
checkpoint outside the repository tree, records the run as `failed-resumable`,
and prints a short redacted summary with the run ID, error class, progress
counters, and a safe resume command.

Examples:

```bash
gonimbus index build --resume-run <run_id>
gonimbus index enrich-with-head --resume-run <run_id>
gonimbus transfer reflow --resume-run <run_id>
```

The resume path validates checkpoint identity and current credentials before it
makes data-plane calls or mutates run state. Resume attempts also claim and
heartbeat a lease so overlapping resumes fail closed instead of racing the same
checkpoint.

Discovery commands now surface resumable state:

- `index list` includes the latest failed-resumable run and a safe resume hint.
- `index stats --runs` includes run IDs, statuses, and failed-resumable counts.
- `index query` warns when the latest run is partial checkpoint state.

## Provider-Dispatched Reflow and Local Backup Pipe

Command code now constructs providers through `internal/providerdispatch`,
which gives `transfer reflow` the same source/destination capability checks
across S3-compatible and local file providers.

The practical operator result is that local directory trees can be selected by
`crawl` and copied into object storage with normal reflow machinery:

```bash
gonimbus crawl --job backup-select.yaml --emit reflow-input \
  | gonimbus transfer reflow --stdin \
      --dest 's3://dest-bucket/landing/'
```

The `crawl --emit reflow-input` records carry `dest_rel_key`, so
`transfer reflow --stdin` preserves arbitrary nested relative paths without
segment rewrite templates. This path supports metadata policy controls,
content-type preservation, storage-class handling, dry-run planning, collision
policies, and JSONL audit records. Local file source paths are redacted in
public output as `file://local/<relative-path>` forms.

Hidden files and dot-directories are skipped by default. Use
`--hidden=include` only after reviewing `--dry-run` output. Symlinks are also
skipped by default; `--symlinks=follow` must be selected explicitly when the
operator wants resolved symlink targets included.

## Resume Collision and Failure-Class Fixes

v0.3.0 includes two reflow correctness fixes that matter for automation around
long-running copies:

- A resumed reflow no longer false-conflicts when the interrupted tail maps to
  already completed, byte-identical destination data. That case is now treated
  as the duplicate-success condition it represents instead of a fatal
  collision, while genuinely different destination data of the same size still
  fails as a conflict.
- Temporary network and transport failures are classified as `TRANSIENT` rather
  than falling through to internal or unknown surfaces.

`TRANSIENT` covers DNS failures, connection resets/refusals/aborts, broken
pipes, mid-transfer EOFs, I/O deadlines, closed network connections, and TLS
handshake timeouts. Reflow failed records use the reason
`transient.network`.

v0.3.0 classifies these failures; it does not add a new automatic retry policy
inside `transfer reflow`. Callers can use the class and existing checkpoint or
resume controls to decide whether and when to retry.

## Dependency Security Gate

The local pre-push hook now stays scoped to changed-file format, lint, and
security checks. Dependency vulnerability enforcement runs in GitHub Actions on
pull requests, pushes to `main`, a daily schedule, and manual dispatch. That
keeps local pushes from requiring `--no-verify` for unscopable dependency
baseline noise while preserving an enforced high-and-critical dependency gate.

The release gate is clean at `fail_on=high`: `golang.org/x/net` is updated to
`v0.55.0`, CI and release workflows use Go `1.26.4`, and no dependency
suppressions are added.

## Library API

This release has additive library-facing changes and no Stable breaks:

- `pkg/provider` adds `ErrCredentialsRefreshFailed` and
  `IsCredentialsRefreshFailed` so provider adapters can mark credential-cache
  refresh failures with a stable sentinel.
- `pkg/output` adds `ErrCodeTransient` so JSONL consumers can distinguish
  temporary network/transport failures from fatal or internal errors.
- Provider dispatch is internal. The new dispatch layer lives under
  `internal/providerdispatch`; the Stable `pkg/uri` and `pkg/provider/s3`
  surfaces are unchanged in v0.3.0.

`pkg/indexstore`, `pkg/opcheckpoint`, and `pkg/reflowstate` remain
Experimental storageful packages.

## Upgrade

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.3.0
```

For local index workflows, the default binary no longer requires CGO. Operators
using remote libsql/Turso indexes should build with `-tags gonimbus_libsql`.

For embedded consumers of the documented Stable packages, v0.3.0 is additive.
Provider implementations may opt into the new credential-refresh sentinel, and
JSONL consumers may switch retry logic to the new `TRANSIENT` class.
