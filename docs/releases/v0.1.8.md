# Gonimbus v0.1.8 Release Notes

**Release Date**: 2026-05-05
**Status**: Index Hub + Workspace Pattern + DX Hardening — Final v0.1.x Release

## Overview

Gonimbus v0.1.8 closes out the v0.1.x line by delivering the publishable / consumable index lifecycle: build an index locally, **publish** it to a hub, **consume** it on another host, and manage hub contents over time. Paired with a documented workspace convention and DX hardening, this is the operational toolchain that production data-acquisition pipelines need.

Key capabilities:

1. **Index Hub** (`index hub` + `index export` + `index hydrate`) — full publish / consume / CRUD lifecycle for index runs against `file://` and `s3://` hubs
2. **Index Query Flags** — `--index-set` for explicit selection, `--output` for streaming results to S3 / file destinations
3. **Workspace Pattern** — convention-first `workspace.yaml` with documented operational flows
4. **Role Catalog** — dataeng role added; attribution policy strengthened
5. **DX Hardening** — pre-push gate scoped to new issues, gosec / golangci-lint debt cleared, guardian hooks retired

## The Problem

The v0.1.7 toolchain could build an index and run probes / reflows against it locally, but had no way to **share** that index with another machine, automate its lifecycle, or publish it for consumers downstream. Operators had to roll their own conventions for "where do indexes live, who can read them, how do we know which run is current."

## The Solution: Hub-Based Index Lifecycle

```bash
# 1. Initialize a hub
gonimbus index hub init --hub s3://my-hub/

# 2. Build an index locally (existing v0.1.7 flow)
gonimbus index build --job manifest.yaml

# 3. Publish to the hub
gonimbus index export --index-set idx_<sha256> --hub s3://my-hub/

# 4. Consume on another host
gonimbus index hydrate --index-set idx_<sha256> --hub s3://my-hub/

# 5. Operate on hub contents
gonimbus index hub ls --hub s3://my-hub/
gonimbus index hub gc --hub s3://my-hub/ --keep 5
```

## New Commands

### `gonimbus index export`

Publishes an index run to a hub:

```bash
gonimbus index export --index-set idx_<sha256> --hub s3://my-hub/
gonimbus index export --index-set idx_<sha256> --hub file:///mnt/hubs/my-hub/
```

**Publish sequence**: `index.db` → `identity.json` → `complete.json` (commit marker) → `latest.json`. Consumers can detect partially-written runs by the absence of `complete.json`.

**Integrity manifest**: `complete.json` contains SHA-256 + size for `index.db` and `identity.json`. Hydrate verifies these before accepting the run.

### `gonimbus index hydrate`

Downloads a published run from a hub:

```bash
# Use the latest.json pointer
gonimbus index hydrate --index-set idx_<sha256> --hub s3://my-hub/

# Or pin to a specific run
gonimbus index hydrate --index-set idx_<sha256> --run-id run_<id> --hub s3://my-hub/
```

- Resolves run via `latest.json` or explicit `--run-id`
- SHA-256 + size verification against `complete.json`
- Rejects uncommitted runs
- Saves `complete.json` to destination for provenance

### `gonimbus index hub` — Full CRUD

| Command      | Purpose                                                                                   |
| ------------ | ----------------------------------------------------------------------------------------- |
| `init`       | Create a new hub root with marker file                                                    |
| `ls`         | List index sets and their runs                                                            |
| `show`       | Show details for a specific index set or run                                              |
| `set-latest` | Advance the `latest.json` pointer (requires committed run)                                |
| `rm-run`     | Remove a specific run (protects `latest` unless `--force`)                                |
| `gc`         | Garbage-collect runs by `--keep N` or `--before DATE`; `--dry-run` and `--json` supported |

### Index Query Flags

```bash
# Explicit index-set selection (resolves prefix or full idx_<64hex>)
gonimbus index query 's3://bucket/prefix/' --index-set idx_da038d8

# Stream results to S3 / file destinations
gonimbus index query 's3://bucket/prefix/' --output 's3://results/query.jsonl'
gonimbus index query 's3://bucket/prefix/' --output 'file:///tmp/query.jsonl'
```

`--index-set` resolves a unique index set when multiple share a base URI; required when auto-selection would be ambiguous.

`--output` lets `index query` participate in pipeline-as-data flows where downstream consumers read JSONL from S3 / file destinations rather than stdout.

## Workspace Pattern

`workspace.yaml` formalizes "where does state live for a body of work":

```yaml
schema_version: gonimbus.workspace.v1
name: my-workspace
tools:
  - gonimbus
description: |
  What this workspace is for.

providers:
  source:
    kind: s3
    bucket_var: SOURCE_BUCKET
  destination:
    kind: s3
    bucket_var: DEST_BUCKET

layout:
  manifests: manifests/
  recipes: recipes/
  indexes: indexes/
  outputs: outputs/
  scripts: scripts/
  notes: notes/
```

The convention is documented in [`docs/user-guide/workspace.md`](../user-guide/workspace.md), covering shard strategies, operational flows (build+publish, hydrate+query, extract+reflow, hub maintenance), rewrite template guidance, and scheduling patterns.

This is convention-first — no code path enforces `workspace.yaml`. Code support catches up in v0.2.x as the convention proves itself in dogfood.

## JSON Schemas

Hub artifacts validate against four schemas published under `schemas/gonimbus/v1.0.0/`:

| Schema                           | Purpose                                   |
| -------------------------------- | ----------------------------------------- |
| `index-hub.schema.json`          | Hub marker (`hub.json` at hub root)       |
| `index-hub-complete.schema.json` | Run commit marker with integrity manifest |
| `index-hub-latest.schema.json`   | Index-set latest pointer                  |
| `index-hub-identity.schema.json` | Index set identity descriptor             |

## latest.json Pointer — Best Effort, v0.1.x

`latest.json` updates use plain `PutObject`: best-effort, last-writer-wins. The schema description and inline comments are explicit about this.

CAS / fail-closed semantics (If-Match / If-None-Match, etag plumbing, provider conditional-put interface) are tracked for v0.2.x. Operators with multiple concurrent exporters should serialize them externally for v0.1.x.

## Role Catalog

- **`dataeng`** role added at `config/agentic/roles/dataeng.yaml` — pipeline operations, manifests, integration testing; updated for v0.1.8 hub/workspace operations
- **Attribution policy** in `AGENTS.md` strengthened: AI-assisted commits MUST use `noreply@3leaps.net`; model-provider domains are rejected to prevent email squatting

## DX Hardening

### Pre-Push Hook Scoping

`.goneat/hooks.yaml` pre-push gate now uses `--new-issues-only --new-issues-base origin/main`, so a docs change doesn't fail-push because of legacy high-severity findings on `main`. Pre-commit (already at `--fail-on=critical`) is unchanged.

### High-Severity Debt Cleared

- 5 gosec G115 / G703 findings annotated with rationale (provably bounded conversions; user-supplied CLI paths)
- 1 golangci-lint QF1012 (`fmt.Sprintf` → `fmt.Fprintf`) in `pkg/manifest/validate.go`

### Guardian Hooks Retired

`.goneat/hooks/pre-commit` and `.goneat/hooks/pre-push` regenerated without `--with-guardian`. The team is on a feature-branch / PR review workflow now, so the browser-intercept approval surface is no longer needed.

## Bug Fixes

### `index hub gc --json` Silent No-Op

`gonimbus index hub gc --json` (without `--dry-run`) emitted the candidate list as JSON and returned **before** the deletion loop. Result: `{"dry_run": false, "removed": [...]}` was reported while artifacts were still in the hub.

**Fix**: routed deletion through the same code path regardless of output mode. JSON output now reports per-run outcomes (`artifacts` count + `error` per candidate, top-level `errors` count). Regression test added (`TestRunIndexHubGC_JSONWithoutDryRunActuallyDeletes`).
