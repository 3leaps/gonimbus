# Gonimbus v0.4.0 Release Notes

**Release Date**: 2026-07-09
**Status**: Durable Index Default

Gonimbus v0.4.0 makes the durable index format the default local and hub-facing
index artifact. Instead of a single growing SQLite `index.db` as the center of
the operator workflow, `index build` publishes a durable-v2 snapshot: sealed
crawl journals, immutable Snappy-Parquet segments, an internal manifest, and
explicit complete/latest markers for hub movement.

SQLite remains a first-class **compatibility path**. Existing `index.db` files
are not rewritten or invalidated. Local query, enrich-head, stats, most doctor
workflows, **`index list`**, and **`index gc`** still use SQLite today — use
`--format sqlite` or `--format both` when you need those commands during the
transition. Under the durable default a plain build produces no `index.db`, so
durable-only sets are not yet enumerated by `list` / `gc`.

## Highlights

- **Durable is the default build format** — `index build` produces durable-v2
  snapshots unless you opt into SQLite compatibility.
- **Export wall relief** — large indexes publish as multiple segment objects
  (plus a small manifest and markers) instead of one multi-gigabyte database
  PUT. Field validation at multi-million object scale showed durable snapshots
  packing into tens-of-megabyte segments while remaining row-level parity clean
  against SQLite on the LIST projection.
- **Dual-format parity** — `--format both` builds SQLite and durable from one
  crawl and emits a machine-readable compare result so operators can trust
  migration before switching consumers.
- **Hub export and hydrate** understand `sqlite-v1` and `durable-v2` markers,
  reject unknown formats, and verify durable digests before trust.
- **Scoped dual-format builds** — `build.scope` works under durable and
  `--format both` with fail-closed coverage equal to the crawl prefix plan.
- **Progress on the default path** — multi-minute durable builds emit crawl
  progress and a segmenting-tail line on stderr so a busy build is not mistaken
  for a hang.

## What operators should do differently

```bash
# Default: durable snapshot (no index.db)
gonimbus index build --job index.yaml

# Need local query / enrich-head / stats / list / gc today?
gonimbus index build --job index.yaml --format sqlite
# or dual-format for parity + SQLite consumers + local inventory
gonimbus index build --job index.yaml --format both

# Export prefers durable when a local durable complete marker exists
gonimbus index export --hub s3://bucket/index-hub/ --index-set idx_...

# Hydrate is format-aware from the hub complete marker
gonimbus index hydrate --hub s3://bucket/index-hub/ \
  --index-set idx_... --dest /tmp/hydrated/
```

### Compatibility notes

- Resume printed as `gonimbus index build --resume-run <run_id>` still works
  under the durable default when `--format` is omitted (SQLite checkpoint path).
- `index export --format auto` (default) selects durable-v2 when a local durable
  complete marker is present for the run; otherwise sqlite-v1.
- Durable hydrate restores `manifest.json` + segments, not `index.db`.
- **`index list` and `index gc` currently enumerate only SQLite index sets**
  (directories with an `index.db`). Durable-only builds are not yet listed —
  use `--format both` (or `sqlite`) if you need them visible locally during the
  transition.

## What green parity means

A green old-vs-new compare result certifies **LIST-projection fidelity** between
SQLite and durable over the same crawl (canonical fields such as relative key,
size, last modified, storage class). It does **not** certify reflow-input
readiness or HEAD-enrichment parity. The compare result carries a
`projection_semantics` block that names what is included and what is excluded.

## Internal-render framing (important)

Durable-v2 in this release is a full-fidelity **internal render** for trusted
operator and pipeline use. It is **not** a reduced-trust, de-identified, or
third-party publication format. Reduced-trust boundary rendering and column
suppression remain a separate future surface. Do not treat a durable hub export
as a disclosure-controlled share format without that later boundary path.

## Upgrade notes

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.4.0
```

1. Install v0.4.0 and run a dual-format build (`--format both`) on a
   representative unit if you currently depend on SQLite consumers.
2. Confirm parity, then switch default durable builds for new runs.
3. Keep `--format sqlite` or `both` for local query/enrichment, inventory
   (`list` / `gc`), and doctor until those commands are durable-aware.
4. Point hub consumers at format-aware hydrate; durable dest has no `index.db`.

## Library consumers

- `pkg/indexbuild` is the durable build engine surface (Experimental embedding
  posture unless otherwise noted in `docs/api-stability.md`).
- Hub consumers should branch on `sqlite-v1` / `durable-v2` markers and reject
  unknown formats.
- `pkg/reflow` Stable surface is unchanged in this cut.
- Downstream index consumers that assumed every build produces `index.db` should
  treat durable as the default and open SQLite only when that format was
  requested or dual-built.
