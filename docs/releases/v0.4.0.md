# Gonimbus v0.4.0 Release Notes

**Release Date**: 2026-07-09
**Status**: Durable Index Default

Gonimbus v0.4.0 is the **index-substrate epoch**: the durable index format becomes
the default local and hub-facing index artifact. Instead of centering the
operator workflow on a single growing SQLite `index.db`, `index build` publishes
a durable-v2 snapshot — sealed crawl journals, immutable Snappy-Parquet segments,
an internal manifest, and explicit complete/latest markers for hub movement.

**Why this matters.** Large inventories hit provider **single-object PUT ceilings**
when published as one database file. Durable packing splits the snapshot into
segment objects (plus a small manifest and markers), so hub export and hydrate
scale past the old monolith wall while keeping row-level LIST-projection parity
against SQLite on validated field runs. The largest individual hub PUT becomes a
segment (typically tens of megabytes under default packing), not the whole
inventory.

SQLite remains a first-class **compatibility path**. Existing `index.db` files
are not rewritten or invalidated. Use `--format sqlite` when a workflow needs
the canonical `index.db` (SQLite-only surfaces such as `query --since-run`,
`stats --prefixes`, and full `--resume-run` recovery). `--format both` does not
produce a canonical `index.db`: it publishes the durable artifact and builds a
run-scoped SQLite projection purely as per-run parity verification.

Operator walkthrough: [Durable Index Format](../user-guide/durable-index.md).

## Highlights

- **Durable is the default build format** — `index build` produces durable-v2
  snapshots unless you opt into SQLite compatibility.
- **Export wall relief** — large indexes publish as multiple segment objects
  (plus a small manifest and markers) instead of one multi-gigabyte database
  PUT. Field validation at multi-million object scale showed durable snapshots
  packing into tens-of-megabyte segments while remaining row-level parity clean
  against SQLite on the LIST projection.
- **Dual-format parity** — `--format both` publishes the durable artifact and
  verifies it against a run-scoped SQLite projection built from the same crawl,
  emitting a machine-readable compare result plus a receipt whose
  `formats_committed` and `verification` fields keep "verified" and "committed"
  as separate claims. Successive `both` builds extend durable lineage
  continuously.
- **Temporal durable compare** — `index compare durable-delta` reports
  added / changed / tombstoned rows between two durable snapshots with
  fail-closed coverage attribution.
- **Hub export and hydrate** understand `sqlite-v1` and `durable-v2` markers,
  reject unknown formats, and verify durable digests before trust.
- **Scoped dual-format builds** — `build.scope` works under durable and
  `--format both` with fail-closed coverage equal to the crawl prefix plan.
- **Progress on the default path** — multi-minute durable builds emit crawl
  progress and a segmenting-tail line on stderr so a busy build is not mistaken
  for a hang.
- **Default segment packing** — 500,000 rows per segment (engine packing lever;
  not operator-facing configuration in this cut).

## What operators should do differently

```bash
# Default: durable snapshot (no index.db)
gonimbus index build --job index.yaml

# Need the canonical index.db (SQLite-only surfaces)?
gonimbus index build --job index.yaml --format sqlite
# Durable publication + per-run SQLite parity verification
gonimbus index build --job index.yaml --format both

# Export prefers durable when a local durable complete marker exists
gonimbus index export --hub s3://bucket/index-hub/ --index-set idx_...

# Hydrate is format-aware from the hub complete marker
gonimbus index hydrate --hub s3://bucket/index-hub/ \
  --index-set idx_... --dest /tmp/hydrated/

# Temporal compare between two durable snapshots
gonimbus index compare durable-delta \
  --before-manifest /path/to/before/manifest.json \
  --before-segments /path/to/before/segments \
  --after-manifest /path/to/after/manifest.json \
  --after-segments /path/to/after/segments
```

### Compatibility notes

- Resume printed as `gonimbus index build --resume-run <run_id>` still works
  under the durable default when `--format` is omitted (SQLite checkpoint path).
- `index export --format auto` (default) selects durable-v2 when a local durable
  complete marker is present for the run; otherwise sqlite-v1.
- Durable hydrate restores `manifest.json` + segments, not `index.db`.
- `index list`, `index stats`, `index doctor`, and `index query` are
  format-aware: durable sets resolve through their verified latest marker, and
  a verified durable latest is preferred over a set-root `index.db` for the
  same set. Use `--format sqlite` when a workflow needs the canonical
  `index.db` artifact itself.

## What green parity means

A green old-vs-new compare result (from `--format both`) certifies
**LIST-projection fidelity** between SQLite and durable over the same crawl
(canonical fields such as relative key, size, last modified, storage class). It
does **not** certify reflow-input readiness or HEAD-enrichment parity. The
compare result carries a `projection_semantics` block that names what is
included and what is excluded. See
[Index Compare Projection v1](../architecture/index-compare-projection-v1.md).

## Internal-render framing (important)

Durable-v2 in this release is a full-fidelity **internal render** for trusted
operator and pipeline use. It is **not** a reduced-trust, de-identified, or
third-party publication format. A durable hub export is not a
disclosure-controlled share format; rendering a durable index for any
reduced-trust or third-party context is out of scope for this format.

## Upgrade notes

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.4.0
```

1. Install v0.4.0 and run a dual-format build (`--format both`) on a
   representative unit if you currently depend on SQLite consumers.
2. Confirm LIST-projection parity, then switch default durable builds for new
   runs and hub traffic.
3. Keep `--format sqlite` for workflows that need the canonical `index.db`
   (`query --since-run`, `stats --prefixes`, full `--resume-run` recovery).
4. Point hub consumers at format-aware hydrate; durable dest has no `index.db`.
5. Read [Durable Index Format](../user-guide/durable-index.md) for the full
   operator map (parity semantics, hub round-trip, SQLite escape hatches).

## Library consumers

- `pkg/indexbuild` is the durable build engine surface (**Experimental** —
  see `docs/api-stability.md`). Callers inject a `pkg/provider` handle, optional
  `CrawlPrefixes` plan input, and path config; the engine does not construct
  providers or import SQLite index-store packages.
- Hub consumers should branch on `sqlite-v1` / `durable-v2` markers and reject
  unknown formats. Programmatic parity consumers should read
  `gonimbus.index.compare_result.v1` `projection_semantics` rather than assuming
  green means reflow-ready.
- `pkg/reflow` remains **Experimental** (no Stable promotion and no Stable
  library API break in this cut).
- Downstream index consumers that assumed every build produces `index.db` should
  treat durable as the default and open SQLite only when that format was
  requested or dual-built. See
  [Library consumers](../library-consumers.md#durable-index-builds-pkgindexbuild).
