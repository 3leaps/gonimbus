# Gonimbus v0.2.2 Release Notes

**Release Date**: 2026-05-26
**Status**: Index Archive Operations and Local-Tree Reflow

## Overview

Gonimbus v0.2.2 sharpens two operator workflows that now sit on the critical
path for high-scale object storage work:

1. **Local-tree reflow** — local directories can be migrated through the same
   audited `transfer reflow` path as object-store sources, with hidden paths
   skipped by default.
2. **Index archive operations** — indexes now retain LIST-derived storage
   class and can be enriched with HEAD-derived archive, restore, and
   content-type fields for filtered candidate rows.
3. **Release toolchain currency** — CI and release builds now use Go `1.26.3`
   and an updated golangci-lint action.

For the comprehensive list of changes, see
[CHANGELOG.md](../../CHANGELOG.md#022---2026-05-26).

## Chapter 1: Local-tree reflow

### The shift

`transfer reflow` now accepts `file://` sources:

```bash
gonimbus transfer reflow 'file:///absolute/source-root/' \
  --dest 's3://bucket/landing/' \
  --rewrite-from '{path}/{file}' \
  --rewrite-to '{path}/{file}' \
  --dry-run
```

The file source path uses the existing reflow engine: rewrite templates,
collision handling, metadata controls, checkpoint/resume, dry-run, and JSONL
audit output. That makes local tree migration an operator workflow rather than
a separate ad hoc copy command.

### Hidden paths

Local-tree reflow skips hidden files and dot-directories by default. Any path
segment beginning with `.` is omitted during traversal, including examples such
as `.env`, `.git/config`, `.DS_Store`, and `nested/.cache/file`.

Use `--hidden=include` only when the destination is expected to receive hidden
paths:

```bash
gonimbus transfer reflow 'file:///absolute/source-root/' \
  --dest 's3://bucket/landing/' \
  --rewrite-from '{path}/{file}' \
  --rewrite-to '{path}/{file}' \
  --hidden=include \
  --dry-run
```

Hidden filtering is not gitignore-aware. Non-hidden generated paths still need
explicit excludes before the first cloud write:

```bash
--exclude 'node_modules/*' \
--exclude 'dist/*' \
--exclude 'target/*' \
--exclude '__pycache__/*' \
--exclude '*.pyc' \
--exclude '*.log'
```

### Local path disclosure

Per-object JSONL output uses `file://local/<relative-path>` instead of exposing
the absolute source root. Checkpoint files and run metadata may still contain
the absolute source root and should be treated as local operational metadata.

See [local tree migration](../user-guide/local-tree-migration.md) for the
operator pattern.

## Chapter 2: Storage class in indexes

Index builds now persist provider storage class values returned during LIST
operations. Query JSONL emits `storage_class` when the provider supplied one:

```bash
gonimbus index query 's3://bucket/prefix/' \
  --storage-class GLACIER,DEEP_ARCHIVE
```

`--storage-class` is repeatable and accepts comma-separated values. Matching is
exact and case-sensitive against the raw provider value. Objects without a
provider-supplied storage class omit `storage_class` from JSONL output and are
not matched by the filter.

The filter is pushed into SQL and composes with canonical-by-ETag query mode,
so storage-class reduction happens before canonical grouping.

## Chapter 3: HEAD enrichment

`index enrich-with-head` caches expensive HEAD-only metadata on an existing
index:

```bash
gonimbus index enrich-with-head idx_da038d8171b4a9ba \
  --storage-class GLACIER,DEEP_ARCHIVE \
  --pattern "**/*.xml" \
  --parallel 32 \
  --state-out enrich-state.jsonl
```

Supported v1 candidate filters are:

- `--storage-class`
- `--pattern`
- `--key-regex`
- `--min-size`
- `--max-size`
- `--include-deleted`

Filters are applied before HEAD calls, so excluded rows do not incur provider
HEAD cost. v1 supports S3-family indexes and rejects unsupported providers
explicitly.

The command writes only HEAD-derived fields:

- `archive_status`
- `restore_state`
- `restore_expiry`
- `content_type`
- `head_enriched_at`

It does not overwrite LIST-derived `storage_class`, size, ETag,
`last_modified`, `last_seen_run_id`, or `deleted_at`. `--resume` skips rows
whose `head_enriched_at` is already set. A full run re-HEADs all candidate rows
and updates only successful HEAD responses.

After enrichment, `index query` can emit the cached fields and filter by
`--enriched-after`:

```bash
gonimbus index query 's3://bucket/prefix/' \
  --storage-class GLACIER,DEEP_ARCHIVE \
  --enriched-after 2026-05-25T00:00:00Z
```

## Chapter 4: Release toolchain

CI and release workflows now pin `actions/setup-go` to Go `1.26.3`, and CI uses
`golangci-lint-action` v2.11.2. This keeps build, scan, and release evidence on
a patched Go 1.26 lane while preserving the module's existing compatibility
floor.

## Upgrade notes

### Operator action

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.2.2
```

Existing object-store reflow and index query workflows remain compatible.
Recommended operator checks:

1. For local-tree reflow, run `--dry-run` first and decide whether hidden paths
   should remain skipped or be included explicitly.
2. Add `--exclude` rules for non-hidden generated paths that should not be
   copied.
3. For archive workflows, run storage-class queries first, then use
   `index enrich-with-head` on the narrowed candidate set.
4. Treat `--state-out` output and local checkpoint data as operational evidence
   that may reveal local paths or object keys.

### Compatibility

No JSONL breaking change is introduced in v0.2.2. New fields are additive and
omitted when unknown. `index enrich-with-head` mutates only HEAD-derived columns
in existing indexes.
