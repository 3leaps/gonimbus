# Gonimbus v0.3.3 Release Notes

**Release Date**: 2026-06-17
**Status**: Adaptive Reflow Concurrency and Reflow Surface Hardening

Gonimbus v0.3.3 makes large `transfer reflow` runs faster to right-size and
safer to trust. `--parallel` becomes a _requested ceiling_ that the engine
adapts within — discovering an endpoint's sustainable rate on its own instead of
making operators guess — while a hard, resource-derived cap keeps even an extreme
`--parallel` from harming the host. Around that headline, the release hardens the
reflow surface: capability-aware collision fallback for S3-compatible stores that
do not honor conditional writes, priority-ordered probe extraction that fails
closed on truncated reads, and classified, credential-redacted error output.

## Highlights

- **Adaptive `transfer reflow --parallel`** — a requested ceiling with a
  resource-capped effective ceiling and default-on throttle-aware AIMD control;
  `--no-adaptive` runs fixed at the effective ceiling.
- **Capability-aware collision fallback** — non-overwrite collision modes fail
  closed to a HEAD/compare path on S3-compatible destinations that do not honor
  `If-None-Match: *`, with a structured warning and audit summary fields.
- **Priority XML XPath probe fallbacks** — ordered `xpath_priority` resolution
  with audit fields, failing closed to quarantine on truncated fallback reads.
- **Reflow error hardening** — compact classified resumable abort causes and
  credential redaction on published provider-error surfaces.
- **S3 auth and transport polish** — hermetic signing coverage for
  profile/default-chain credential paths and opt-in S3 transport tuning from
  transfer reflow.

## Adaptive Reflow Concurrency

`transfer reflow --parallel` no longer pins a fixed worker count. It now names a
**requested ceiling**, and the run does two things with it:

1. **Resolves an effective ceiling** — `min(--parallel, resource cap)`, where the
   resource cap is derived from available memory and the process file-descriptor
   limit. Because the cap is computed at startup, independent of endpoint
   feedback, even `--parallel 9385` against a never-throttling endpoint settles
   at a value the host can survive.
2. **Adapts within that ceiling** — adaptive mode (on by default) uses
   additive-increase / multiplicative-decrease, the same shape as TCP congestion
   control: start cautiously, halve on a throttle signal, then step back up one
   increment at a time after a sustained clean streak. The concurrency a run
   converges to is, in effect, the **empirically discovered sustainable rate**
   for that endpoint and host.

```bash
# Ask for up to 256 workers; the engine settles where the endpoint + host allow.
# (reflow-input.jsonl carries dest_rel_key per record, e.g. from `content probe --emit reflow-input`)
gonimbus transfer reflow --stdin \
  --dest 's3://dest-bucket/reflowed/' \
  --parallel 256 < reflow-input.jsonl

# Fixed mode: run at the resource-capped effective ceiling, no adaptation.
gonimbus transfer reflow --stdin \
  --dest 's3://dest-bucket/reflowed/' \
  --parallel 64 --no-adaptive < reflow-input.jsonl
```

The clamp is never silent. When the effective ceiling is below the requested
value, the run reports **why** through `concurrency_ceiling_reason` (`requested`,
`resource_capped:memory:<source>`, or `resource_capped:fd`) plus a stderr notice,
so a lower-than-requested concurrency is explained rather than inferred from
throughput. Connection-error bursts **freeze** the ramp rather than cutting it (a
network outage is not congestion), and retries stay inside the concurrency budget
so a throttle storm cannot amplify load.

Every run and summary record carries additive `concurrency_*` audit fields —
including `concurrency_ceiling_requested`, `concurrency_ceiling_effective`,
`concurrency_ceiling_reason`, `concurrency_final`, `concurrency_max_active`, and
`concurrency_throttle_backoffs`. Reading them together — _requested 256 →
effective 256 → final 48, 6 throttle backoffs_ — tells you the endpoint's
sustainable rate and whether the host resource cap, not the endpoint, was the
limiting factor.

On S3-compatible destinations, reflow also opts into HTTP transport sizing
derived from the **effective** ceiling, so high concurrency reuses connections
instead of paying a per-request TCP+TLS handshake tax. The tuning is scoped to
the reflow path; `inspect`, `crawl`, and other commands keep the SDK default
transport.

See [`docs/user-guide/reflow.md`](../user-guide/reflow.md#adaptive-parallelism)
for the CLI-facing notes and
[`docs/user-guide/concurrency-and-throughput.md`](../user-guide/concurrency-and-throughput.md)
for the provider-generalized operator model.

## Capability-Aware Collision Fallback

Not every S3-compatible store honors `If-None-Match: *`. On a store that silently
ignores it, a no-overwrite collision mode could previously degrade to an
unguarded overwrite without telling the operator. v0.3.3 closes that gap: before
relying on conditional writes, reflow runs a semantic IfAbsent probe with
tri-state reporting — `honored`, `not_honored`, or `inconclusive` — and **fails
closed to a HEAD/compare fallback** when the destination does not honor the
precondition or the probe is inconclusive.

The fallback covers `skip-if-duplicate`, `fail`, `quarantine`, and
`overwrite-if-source-newer`; an intentional `overwrite` remains unconditional.
Reflow emits a structured warning plus terminal `gonimbus.reflow.summary.v1`
fields for the IfAbsent probe status, fallback activation, and degraded-path
object count — so CI and audit consumers can see exactly when a destination's
conditional-write support was the limiting factor.

## Priority Probe Fallbacks

`content probe` `xml_xpath` extractors now accept `xpath_priority`: an ordered
list of fallback tags resolved highest-priority-first. When the primary tag is
absent, the extractor walks the priority list and records which fallback won
through audit fields — `resolved_priority`, `resolved_xpath`, `truncated_fallback`,
and `truncated_fallback_count`.

Priority resolution fails closed. A lower-priority value observed before the
`max_bytes` or fixed-window read boundary may be an artifact of a truncated read
rather than a genuine match, so it routes to quarantine by default instead of
being accepted as a final normal-routing match. The audit fields make a
truncation-driven fallback visible rather than silent.

## Reflow Error Classification and Redaction

Two changes make reflow failures both more machine-readable and safer to publish:

- **Classified resumable abort causes.** A resumable `transfer reflow` abort now
  emits a compact, classified cause on the operation error record, so automation
  around long-running copies can branch on the failure class rather than parsing
  free-text.
- **Credential redaction on published surfaces.** Published reflow error and
  warning messages redact credential material from provider URLs — every URL
  query value is redacted by default — so signed-URL and other credential-bearing
  provider errors do not leak into JSONL output, logs, or CI artifacts.

## S3 Auth and Transport Polish

- **Signing coverage.** Profile-based and SDK default-chain environment
  credential paths now have hermetic SigV4 signing regression coverage alongside
  the existing static-key tests, so all three credential-resolution paths are
  verified without network access.
- **Opt-in transport tuning.** `transfer reflow` sizes the S3 client's HTTP
  connection pool from the resource-capped effective ceiling (see _Adaptive
  Reflow Concurrency_), and the new `pkg/provider/s3.Config` fields expose the
  same controls to embedders.

## Library API

This release has additive Stable API changes and no Stable breaks:

- `pkg/provider/s3.Config` adds optional `MaxIdleConnsPerHost` and
  `MaxConnsPerHost` HTTP transport sizing fields. Zero values preserve the AWS
  SDK defaults; `transfer reflow` uses the fields to opt into resource-capped S3
  client tuning without changing unrelated provider construction paths.

## Upgrade

```bash
go install github.com/3leaps/gonimbus/cmd/gonimbus@v0.3.3
```

CLI workflows remain compatible with v0.3.2. Existing `transfer reflow --parallel`
invocations keep working — the value is now interpreted as a requested ceiling,
adaptive control is on by default, and `--no-adaptive` restores fixed-concurrency
behavior at the resource-capped effective ceiling. Package-manager installation
follows the Homebrew and Scoop publishing flow established for v0.3.2:

```bash
brew install 3leaps/tap/gonimbus
scoop install gonimbus
```
